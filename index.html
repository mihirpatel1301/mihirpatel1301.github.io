Identity, Access & Governance

Microsoft Entra ID
formerly Azure Active Directory (AAD)
Identity SSO Governance
What It Is
Cloud-native identity & access management (IAM) service. Acts as the directory for users, groups, devices, and applications. Supports OAuth 2.0, OIDC, SAML 2.0.
Key Features
User & Group management (Members, Guests, Dynamic Groups)
App Registrations & Enterprise Applications (SSO)
Conditional Access Policies (MFA enforcement by location/risk)
Identity Protection (risky sign-in detection, token replay)
Hybrid Identity: Password Hash Sync / Pass-through Auth / Federation
B2B Guest Access & B2C (consumer identity)
Real-World Use Cases
Fintech: Enforce MFA for payment portal logins from untrusted locations
Enterprise: SSO across 200+ SaaS apps (Salesforce, ServiceNow, SAP)
Hybrid AD: Sync on-prem AD DS to cloud for unified identity
Vendor onboarding: Grant B2B guest access to internal SharePoint
⚡ Exam Tips
Know the difference between Cloud-Only, Synced (AAD Connect), and Federated users. PHS syncs password hashes; PTA uses on-prem agents. Conditional Access requires P1 license; Identity Protection needs P2.

Azure RBAC
Role-Based Access Control
Permissions Least Privilege
What It Is
Authorization system built on Azure Resource Manager. Defines who (security principal), what (role definition with Actions/NotActions), and where (scope: MG → Sub → RG → Resource).
Built-in Roles to Know
Owner — full access + delegate access
Contributor — full resource CRUD, no RBAC management
Reader — read-only on all resources
User Access Administrator — manage access only
Network Contributor / VM Contributor — scoped roles
Real-World Use Cases
Give DevOps team Contributor on a DEV resource group only
Grant a managed identity read-only access to a Key Vault
Assign Reader role to auditors at subscription scope for cost review
⚡ Exam Tips
RBAC is additive; assignments from broader scopes cascade down. Deny assignments override allow. Custom roles require Actions (allowed), NotActions (excluded from allow), and optional DataActions for storage/blobs.

Azure Policy
Governance & Compliance Enforcement
Compliance Enforce
What It Is
Rule-based governance service that evaluates resources against organizational standards. Can Audit, Deny, Append, Modify, or Deploy-if-not-exists.
Real-World Use Cases
Enforce all VMs must be in approved regions (India South, India Central)
Require mandatory tags (CostCenter, Environment) on all resources
Deny creation of public IP addresses in production subscription
Auto-deploy diagnostic settings to all new storage accounts (DeployIfNotExists)
PCI-DSS initiative: bundle 30+ policies into one compliance set
⚡ Exam Tips
Policy ≠ RBAC. Policy controls what resources can look like; RBAC controls who can do what. Initiatives = policy sets. Remediation tasks fix non-compliant existing resources (needs Managed Identity with Contributor role).

Azure Resource Manager
ARM — Deployment & Management Layer
InfrastructureIaC
What It Is
The unified management layer for Azure. Every Azure action (Portal, CLI, PowerShell, REST API) goes through ARM. ARM handles authentication, authorization (RBAC), and routes requests to resource providers.
Key Concepts
ARM Templates (JSON) — declarative IaC. Define desired state; ARM handles idempotent deployments
Bicep — domain-specific language that compiles to ARM JSON. Simpler syntax, native Azure support
Resource Groups — logical containers; share lifecycle, region, and RBAC boundaries
Resource Providers — Microsoft.Compute, Microsoft.Network, Microsoft.Storage — each manages resource types
Tags — key-value metadata on resources; used for cost allocation, automation, governance
Locks — CanNotDelete / ReadOnly — protect critical resources from accidental changes
Real-World Use Cases
Deploy entire 3-tier app (VNet + VMs + SQL) via one ARM/Bicep template in CI/CD pipeline
Apply ReadOnly lock on production VNet to prevent accidental subnet deletion
Enforce resource group tagging via ARM policy + DeployIfNotExists for audit compliance
⚡ Exam Tips
ARM deployments are idempotent — run same template twice = same result. Resource Group deletion deletes ALL resources inside (important!). Locks are inherited from parent scope. CanNotDelete allows updates; ReadOnly prevents all writes including restarts.

Azure Storage Account
Blob · File · Queue · Table
BlobFilesQueue
Service Types
Blob Storage — unstructured data (images, backups, video, logs). Block / Append / Page blobs
Azure Files — fully managed SMB/NFS file shares. Mount as Z: drive
Queue Storage — async message queuing (max 64KB/message)
Table Storage — NoSQL key-value pairs (cheaper than Cosmos DB)
Replication Options (exam critical)
LRS — 3 copies in 1 datacenter (cheapest)
ZRS — 3 copies across 3 AZs (99.9999% durability)
GRS — LRS + async copy to paired region
GZRS — ZRS + async copy to paired region (best)
RA-GRS / RA-GZRS — read access to secondary region
Access Tiers & Lifecycle
Hot → Warm → Cool → Cold → Archive (increasing latency, decreasing cost)
Lifecycle policy: move blobs based on age (last modified / last accessed)
Archive tier: rehydrate takes up to 15 hours (standard) or 1 hour (high priority)
Security & Access
SAS tokens: Account SAS / Service SAS / User Delegation SAS (best practice)
Stored Access Policies — revoke SAS without regenerating key
Private Endpoints — access storage over VNet private IP
Firewall rules — restrict to specific VNets or IP ranges
Real-World Use Cases
Payment portal: store transaction receipts as Append Blobs (immutable append)
Finance app: mount Azure Files share for legacy app expecting UNC path \\server\share
Data lake: ADLS Gen2 = Storage Account + HNS enabled (hierarchical namespace)
⚡ Exam Tips
StorageV2 (General Purpose v2) is recommended for everything. Standard vs Premium (SSD-backed) is account-level, not per-blob. Soft delete, versioning, and point-in-time restore require blob service settings to be enabled. ADLS Gen2 uses Storage Account with HNS=true.

Azure File Sync
Hybrid File Cache
HybridSMB
What It Is
Syncs on-prem Windows Server file shares with Azure Files. Acts as a distributed cache — frequently used files stay local, cold files are tiered to cloud (cloud tiering).
Architecture Components
Storage Sync Service (Azure resource) — top-level container
Sync Groups — defines the topology (1 cloud endpoint + N server endpoints)
Azure File Sync Agent — installed on Windows Server 2012 R2+
Cloud Tiering — frees local disk by storing rarely-used files in Azure
Real-World Use Cases
Branch office file server migration: centralize to Azure, keep local cache
Disaster recovery: secondary server re-syncs from Azure Files quickly
Multi-site collaboration: all branches sync from the same Azure Files share
⚡ Exam Tips
Cloud tiering uses reparse points — stub files. Server endpoint can be a volume, folder, or DFS namespace folder. Maximum 100 server endpoints per Storage Sync Service.

Azure Files
Fully Managed SMB/NFS Cloud Shares
SMBNFSHybrid
What It Is
Fully managed cloud file shares accessible via SMB 3.x (Windows/Linux/macOS) and NFS 4.1 (Linux). Mount directly to VMs or on-prem servers. Replaces traditional Windows File Servers.
Key Features
SMB Shares — mount as drive letter (Z:) on Windows, /mnt on Linux. Max 100 TB/share
NFS Shares — Linux-native mount. Requires Premium tier (SSD-backed), no public access
Azure File Sync — sync on-prem servers with Azure Files for hybrid caching
Snapshots — share-level incremental snapshots for point-in-time restore
Soft Delete — recover accidentally deleted shares within retention window
Identity-based auth — Entra ID (Kerberos) or on-prem AD DS for SMB access
Tiers
Transaction Optimized — HDD, frequent small I/O. Default standard tier
Hot — HDD, general purpose access patterns
Cool — HDD, infrequent access (online archival)
Premium — SSD. Low latency <1ms. Required for NFS and high I/O workloads
Real-World Use Cases
FSLogix profile containers for Azure Virtual Desktop — store user profiles on Azure Files Premium
Legacy app migration: app expects \\fileserver\share UNC path → mount Azure Files share
Multi-server config file share: all web servers read shared config from one Azure Files mount
Branch office decommission: migrate on-prem file server to Azure Files + File Sync
⚡ Exam Tips
SMB requires port 445 outbound — often blocked by ISPs. Use Azure VPN or Private Endpoint as a workaround. Premium shares are provisioned size (pay for what you allocate). Standard shares are pay-per-use. NFS shares cannot be mounted from on-prem without VPN/ExpressRoute.

Azure Container Instances
ACI — Serverless Container Hosting
ContainersServerless
What It Is
Run Docker containers on-demand without managing VMs or Kubernetes clusters. Fastest way to get a container running in Azure — start in seconds. Billed per second of CPU and memory consumed.
Key Features
Per-second billing — no idle cost if container stops
Container Groups — multi-container pod sharing network namespace and storage
VNet integration — inject container group into a subnet (private ACI)

ACI containers are not meant for production workloads needing HA — use AKS for that. Container Groups share a public IP and can expose multiple ports. ACI supports confidential containers (hardware-based isolation). Restart policy: Always / OnFailure / Never.

Network Security Groups
NSG — Layer-4 Firewall
FirewallL4
How NSGs Work
Stateful L4 firewall — allow/deny by IP, port, protocol
Apply to subnet (recommended) or NIC (more granular)
Rules evaluated by priority (100-4096); lower = higher priority
Default rules: AllowVNetInBound, AllowAzureLoadBalancerInBound, DenyAllInBound
Service Tags: VirtualNetwork, AzureLoadBalancer, Internet, AzureCloud, Storage
ASGs (Application Security Groups) — group VMs by role, not IP
Real-World Use Cases
Allow TCP 443 inbound from Internet to web subnet; deny all else
Allow SQL 1433 from AppServers ASG to DBServers ASG only
Block outbound Internet from DB subnet (DenyAllOutbound rule)
⚡ Exam Tips
NSG at subnet processes before NIC-level NSG for inbound; NIC processes before subnet for outbound. Both NSGs must allow traffic. NSG Flow Logs → Log Analytics for traffic analysis.

Azure Load Balancer
L4 TCP/UDP Traffic Distribution
L4HASNAT
Architecture Components
Frontend IP Configuration — Public IP (internet-facing) or Private IP (internal). Multiple frontends per LB.
Backend Pool — VMs or VMSS instances receiving traffic. Supports NIC-based or IP-based pools.
Load Balancing Rules — maps frontend:port → backend:port. Uses 5-tuple hash (src IP, src port, dst IP, dst port, protocol) by default.
Health Probes — TCP / HTTP / HTTPS probes. Unhealthy instances are removed from rotation. Probe interval default: 15s, threshold: 2 failures.
Inbound NAT Rules — map specific frontend port → specific backend VM port (e.g., port 5001 → VM1:3389 for RDP).
Outbound Rules — configure SNAT pool for private VMs needing internet access. Prevents SNAT port exhaustion.
HA Ports Rule — single rule balances ALL TCP+UDP ports (Standard ILB only). Used for NVA (network virtual appliance) clusters.
Public vs Internal (ILB)
Public LB — frontend is a public IP. Internet → LB → backend VMs. Used for web tier.
Internal LB (ILB) — frontend is a private VNet IP. Used between app tiers (app servers → DB cluster). Never exposed to internet.
SKU Comparison: Basic vs Standard
Basic — free, no SLA, no AZs, no HTTPS probe, open by default (no NSG needed), max 300 instances
Standard — paid, 99.99% SLA, zone-redundant, HTTPS probe, closed by default (NSG required), 1000 instances, secure by default
⚠️ Basic LB is being retired — always use Standard for new deployments
Session Persistence Options
None (5-tuple hash) — default, best distribution, stateless apps
Client IP (2-tuple) — same client always hits same backend
Client IP + Protocol (3-tuple) — same client+protocol hits same backend
Real-World Use Cases
FinCore: Public LB (port 443) → 3 web VMs in VMSS across AZs for payment portal
ILB for SQL Server AlwaysOn AG — listener IP is ILB frontend, routes to primary replica
Outbound Rules: give 10 backend VMs a shared pool of 64K SNAT ports for internet egress
HA Ports on ILB: Palo Alto NVA cluster — all traffic passes through firewall VMs
Inbound NAT: jump to specific VMs — LB:5001→VM1:3389, LB:5002→VM2:3389
⚡ Exam Tips
Standard LB is zone-redundant by default; zone-pinned by choice. Standard LB is "secure by default" — traffic blocked unless NSG explicitly allows. HA Ports rule = protocol:All, port:0 — exclusive to ILB. You CANNOT mix Basic and Standard LB in the same availability set. Floating IP (Direct Server Return) is required for SQL AlwaysOn AG listeners.

Azure Traffic Manager
Global DNS-Based Load Balancing
GlobalDNSRouting
How It Works
DNS-based global traffic routing — Traffic Manager returns the IP of the "best" endpoint based on routing policy
No data passes through Traffic Manager — only DNS responses. Client connects directly to resolved endpoint.

Routing Methods (Exam Critical)
Priority — Primary + failover endpoints. Send all to primary; failover if unhealthy. DR scenario.
Weighted — distribute traffic by weight (70/30 split). A/B testing, gradual migration.
Performance — route to endpoint with lowest latency for the client. Global user base → nearest region.
Geographic — route based on client geographic location. EU users → EU endpoint. Data residency / GDPR.
Multivalue — return multiple healthy endpoints in DNS response (IPv4 only). Client picks one.
Subnet — route based on client IP subnet ranges. ISP-level or office-level routing.
Nested Profiles
Combine routing methods: Performance (outer) + Priority (inner per region)
Route to nearest region, then within that region use priority for DR

Traffic Manager is NOT a proxy — data doesn't flow through it. TTL is typically 60-300s — failover takes TTL time. Traffic Manager supports external endpoints (non-Azure). Azure Service Bus
Enterprise Messaging & Pub/Sub Broker
MessagingPaaSBroker
What It Is
Fully managed enterprise message broker with queues and publish-subscribe topics. Decouples applications and services from each other — senders and receivers don't need to be online at the same time. Built for mission-critical workloads requiring guaranteed delivery, ordering, and transactional consistency.
Two Entity Types
Queues — point-to-point (P2P). One sender, one receiver. Message stays in queue until consumed and acknowledged. Supports FIFO via message sessions.
Topics & Subscriptions — publish-subscribe (pub/sub). One sender, multiple independent subscribers. Each subscription gets a copy of every message. Support filter rules (SQL-like) per subscription.
Key Features (Deep Dive)
Dead-Letter Queue (DLQ) — built-in sub-queue that catches messages that cannot be delivered or processed after max delivery count (default: 10). Inspect poison messages without losing them.
Message Sessions — group related messages with a session ID for strict FIFO ordering. Exactly one processor owns a session at a time. Required for order-of-operations guarantees.
Scheduled Messages — enqueue a message but make it visible only at a future time. Useful for delayed processing, reminders, SLA escalations.
Message Deferral — receiver defers a message (saves for later processing) without removing it from the queue. Useful when out-of-order messages arrive.
Transactions — atomic operations across multiple Service Bus entities. Send + receive + complete in a single transaction — critical for financial systems.
Duplicate Detection — detect and discard duplicate messages using MessageId within a configurable time window. Prevents double processing of payments.
Auto-forwarding — automatically forward messages from one queue/subscription to another. Build pipeline chaining without custom code.
Geo-Disaster Recovery (GeoDR) — namespace-level pairing between primary and secondary regions. Failover in seconds; Premium tier only.
Tiers Comparison
Basic — Queues only. Max 256 KB message. No topics, no sessions, no DLQ on topics. Dev/Test only.
Standard — Queues + Topics. Max 256 KB. Shared capacity. Variable throughput. Pay-per-million operations.
Premium — Dedicated capacity (Messaging Units). Max 100 MB message. VNet integration, Private Endpoints, GeoDR, JMS 2.0. Production recommended.
Protocols & SDKs
AMQP 1.0 — primary protocol. Efficient binary, supports sessions and transactions. Preferred for .NET, Java, Python SDKs.
HTTPS / REST — for simple send/receive without persistent connections. Works through firewalls.
JMS 2.0 — Java Message Service support (Premium tier). Drop-in for legacy Java JMS apps migrating to cloud.
SDKs: .NET, Java, Python, JavaScript/TypeScript, Go — all available via Azure SDK
Service Bus vs Storage Queue
Service Bus — enterprise: FIFO sessions, transactions, DLQ, topics, 100 MB messages, duplicate detection. Use for financial/order workflows.
Storage Queue — simple: 64 KB max, no ordering, no built-in DLQ, no pub/sub. Use when you need cheap, massive-scale simple queuing and are already using Storage Account.
Rule: if you need guaranteed ordering, at-most-once delivery, transactions, or pub/sub → always choose Service Bus.
Real-World Use Cases
Payment order processing (FinCore) — order service publishes to Service Bus Topic → payment, inventory, notification services each subscribe independently. Duplicate detection prevents double-charge on retry.
FIFO trade execution — sessions ensure buy/sell orders for the same stock symbol are processed in strict sequence by a single worker
Loan approval workflow — scheduled messages trigger escalation after SLA breach (48 hrs); DLQ catches failed credit-check messages for ops review
Microservices decoupling — 12 downstream services consume from same Topic via independent subscriptions with SQL filters (e.g., only high-value transactions > ₹1L)
Legacy migration — on-prem MQ Series / RabbitMQ replaced by Service Bus Premium with JMS 2.0 — zero code change in Java consumers
Architecture Pattern — Topic with Filter Subscriptions
Publisher sends all transaction events to one Topic
Subscription "HighValue" — SQL filter: amount > 100000 → routed to fraud detection service
Subscription "AllTxn" — no filter → routed to audit log service
Subscription "INR" — SQL filter: currency = 'INR' → routed to RBI reporting service
Each subscriber processes independently — failure in one does not affect others
⚡ Exam Tips
Service Bus Namespace is the top-level container (like a server). Queues and Topics live inside it. Premium tier required for VNet integration, Private Endpoint, and GeoDR. Max lock duration for message processing: 5 minutes (extendable via RenewMessageLock). DLQ path format: queuename/$DeadLetterQueue. Message TTL default: 14 days (configurable). At-most-once delivery = "Receive and Delete" mode (no acknowledgement needed). At-least-once = "Peek-Lock" mode (explicit Complete/Abandon).
